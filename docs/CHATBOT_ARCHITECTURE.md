# Chatbot Architecture

The chatbot is a local educational assistant and deterministic command automation layer. It helps users understand graph algorithms and can trigger supported app actions without replacing the main UI.

## 1. Chatbot Purpose

The chatbot exists to:

- explain graph algorithm concepts,
- compare algorithms,
- answer common learning questions,
- describe the current algorithm step,
- parse simple English and Albanian commands,
- trigger supported algorithm runs through a controlled action layer.

It is not currently a networked AI assistant. It is rule-based and deterministic.

## 2. Command Parser

Command parsing is handled by `src/lib/chatbot-command-parser.ts`.

The parser receives:

- raw user text,
- current available node labels.

It returns a typed action such as:

- `RUN_ALGORITHM`
- `SET_START_NODE`
- `SET_END_NODE`
- `CLEAR_RESULT`
- `RESET_VISUALIZATION`
- `EXPLAIN_ONLY`
- `UNKNOWN`

## 3. Intent Detection

The parser uses regular expressions and normalization to detect:

- algorithm names and common misspellings,
- execution verbs,
- shortest-path phrases,
- MST phrases,
- comparison questions,
- explanation questions,
- reset/clear commands,
- start and end node labels.

The parser intentionally keeps explain-only questions separate from commands that should mutate app state.

## 4. Supported Commands

Examples of supported command styles:

- `Run Dijkstra from A to F`
- `Find the shortest path from A to E`
- `Execute BFS starting from node 1`
- `Use Bellman-Ford for negative weights`
- `Create the minimum spanning tree using Kruskal`
- `Run Prim from B`
- `Clear the current result`
- `Reset the algorithm visualization`

Albanian equivalents are also supported for common phrasing.

## 5. Albanian Support

The parser includes Albanian command patterns for:

- running algorithms,
- shortest path requests,
- MST requests,
- BFS/DFS phrasing,
- explanation questions,
- complexity questions,
- clear/reset actions.

The response layer also includes Albanian educational explanations and fallback messages.

## 6. English Support

English support covers:

- direct algorithm execution commands,
- shortest path requests,
- comparison prompts,
- negative-weight questions,
- complexity questions,
- common term explanations,
- current-step explanations.

## 7. Action Execution Layer

Action execution is handled by `src/lib/chatbot-action-handler.ts`.

The action handler receives a parsed action and a controlled context containing:

- graph nodes,
- graph edges,
- selected start/end nodes,
- setters for start/end nodes,
- `runAlgorithm`,
- `clearResult`,
- current language.

It validates:

- graph is not empty,
- required start/end node exists,
- requested node labels exist,
- MST algorithms are not run on directed graphs,
- Dijkstra/A* are not run with negative weights.

Only after validation does it call the app's existing algorithm execution flow.

## 8. Response Generation Layer

Educational responses are generated by `src/lib/chatbot-responses.ts`.

This layer provides:

- algorithm explanations,
- complexity answers,
- comparison answers,
- term definitions,
- common mistakes,
- current-step explanations,
- fallback suggestions.

Responses are local and deterministic.

## 8.1 Algorithm Mentor Layer

The educational track now includes a deterministic **Algorithm Mentor**
(`src/lib/mentor/`). It is wired in through a single early call inside
`generateChatbotResponse`:

```ts
const mentorReply = generateMentorResponse(trimmed, ctx);
if (mentorReply) return mentorReply;
```

`generateMentorResponse` classifies a question into one of five mentor intents
and delegates to a matching pure engine, or returns `null` so the existing
handlers run unchanged:

- `recommend` — which algorithm to use, reasoned from problem properties
  (weighted / directed / negative / task). Engine: `recommendation-engine.ts`.
- `recommend-graph` — what to use for the currently loaded graph, by inspecting
  graph state (read-only, never executes). Engine: `graph-analysis.ts`.
- `compare` — structured purpose / complexity / strengths / weaknesses /
  use-cases for two algorithms. Engine: `comparison-engine.ts`.
- `why` — structured purpose / strengths / weaknesses / use-cases for one
  algorithm. Engine: `why-engine.ts`.
- `step-why` — why the current execution step happened (e.g. "why did Kruskal
  reject this edge?"), grounded in `currentStep`. Engine:
  `step-explanation-engine.ts`.

A bilingual knowledge base (`algorithm-knowledge.ts`) is the single source of
truth. The mentor is fully deterministic and local — no LLM, no network. It
never touches the command parser or action handler, so the automation track is
unchanged. A small **Assistant ⇄ Mentor** mode toggle in `GraphChatbot.tsx`
adjusts the welcome, subtitle and suggested questions; mentor capabilities work
in both modes.

See `ALGORITHM_MENTOR_ARCHITECTURE.md` for full details.

## 9. Safety Boundaries

Current safety boundaries:

- The chatbot can only execute predefined app actions.
- It cannot create arbitrary code.
- It cannot modify files.
- It cannot call external services.
- It cannot run shell commands.
- It cannot bypass app validation.
- It cannot directly mutate graph state except through the provided controlled context.
- Unknown or ambiguous commands fall back to educational responses.

## 10. Future AI Opportunities

OpenAI or ChatGPT integration could be added later without replacing the deterministic chatbot.

Recommended future approach:

- Keep the current command parser for safety-critical app actions.
- Use an AI model only for richer explanations, tutoring, and question answering.
- Route all action requests through the existing deterministic parser and action handler.
- Provide the model with read-only graph context, such as node labels, edge count, selected algorithm, and current step.
- Ask the model to explain, not directly execute.
- Use structured outputs only for suggested intents, then validate those intents with the current action layer.
- Preserve English and Albanian support.

This hybrid design would keep the app safe and predictable while improving the educational assistant experience.
